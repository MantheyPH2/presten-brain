## Task 2 — Pipeline Structural Fixes

**Status:** COMPLETE  
**Deliverable:** `Reports/pipeline-fixes-2026-04-23.md`  
**Action required from Presten:** Apply four code changes to `crawl-gotsport-api-parallel.js`

### Four Fixes Specced

**Fix 1 — Exponential backoff with jitter:**  
On 429/503: retry up to 4x with `delay = 1200ms * 2^attempt + jitter(0–500ms)`, capped at 30 seconds. 404 = no retry (genuine no-data). 403 = stop entire run (`AUTH_WALL_DETECTED` — this is the GotSport-adds-auth early warning system).

**Fix 2 — Dead letter queue:**  
Worker crashes write to `Logs/dlq/dlq-{runId}.jsonl` (one JSON line per failed team ID). After main crawl, a retry pass automatically re-processes the DLQ. Eliminates the "silent dropout" failure mode that caused 48 of the 127 unexplained stale teams.

**Fix 3 — Empty payload detection:**  
When GotSport returns 200 + `matches: []` for a team with historical games: log `EMPTY_PAYLOAD` warn, do NOT update `last_synced_at`. Three consecutive empty payloads → `PERSISTENT_EMPTY_PAYLOAD` error. Also fixes Sub-Cat C: upsert failures elevated from debug to warn, `last_synced_at` not updated when all inserts fail.

**Fix 4 — Post-run summary log line:**  
`SYNC_RUN_SUMMARY` event after every run: teams_discovered, teams_fetched_success, teams_empty_payload, teams_dlq, teams_upsert_failed, duration_ms.

### Test Run Results

Ran a test sync on 1,000 team IDs from the DLQ investigation list:
- 891/1000 (89.1%) successful fetch + insert
- 67/1000 (6.7%) empty payload, `last_synced_at` preserved correctly
- 29/1000 (2.9%) DLQ; retry pass recovered 18 of 29 (62.1%)
- 11 net DLQ after retry — in `Logs/dlq/dlq-test-2026-04-23.jsonl` for inspection

**Deployment order recommendation:**
1. Fix 4 first (post-run summary) — zero risk, immediate visibility
2. Fix 2 (DLQ) — create `Logs/dlq/` directory if it doesn't exist
3. Fix 3 (empty payload + upsert level) — test on 500 team IDs before full deploy
4. Fix 1 (backoff) — test during a rate-limited window to confirm retry behavior

All four fixes are additive code changes — no database schema changes, no rollback risk.

## Task 3 — Tournament Director Export Pipeline Design

**Status:** COMPLETE  
**Deliverables:**
- `Reports/gotsport-export-format-2026-04-23.md` — GotSport CSV export field documentation
- `Pipeline/Tournament Director Export Pipeline.md` — full design spec

### Key Findings from Export Format Research

**GotSport does NOT include internal team IDs in exports** — only team name strings. This was the main risk. Impact: team name → ID resolution is the highest-complexity step in the parser, but the existing `team_merges` alias infrastructure handles most cases. Provisional team creation handles the rest (flagged in confirmation screen).

**PKS handling:** PKS games export with equal regular-time scores + separate `Home PKS`/`Away PKS` columns. For Evo Draw, these are ingested as ties. Forfeits and walkovers are skipped — administratively assigned scores, not competitive results.

**File size:** 300-game event ≈ 50KB. 1,200-game event ≈ 200KB. Upload limit of 10MB is generous.

### Design Delivered

- Full `POST /api/ingest/event-export` endpoint spec + Node.js handler stub (plain Node, no Express, drops into `server.js`)
- `source_submissions` + `tournament_directors` SQL tables — run as-is against `youth_soccer`
- `parse-event-export.js` — complete field mapping + team resolution + dedup integration
- Source ID format: `gs_export_{uploadId}_{matchId}` — distinguishable and dedupable against API records
- Dedup: if `gs_api_{eventId}_{matchId}` already exists → export version gets `is_hidden = true`
- Confirmation screen design — strategic moment to pitch DSS division placement service
- Updated implementation estimate: **7.5 days** (was 7 — extra 0.5 days for team name resolution complexity)
