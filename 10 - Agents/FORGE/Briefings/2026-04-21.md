First active day for FORGE on the Evo Draw platform. ---

## Tasks Completed

### 1. Pipeline Documentation Review
Reviewed all docs under `Pipeline/` including the GotSport API integration spec, the nightly sync scheduler config, and the ETL transformation maps. The architecture is sound overall — event ingestion, normalization, and roster hydration are cleanly separated. A few undocumented assumptions in the transformation layer (notably around bye-week game handling) were noted inline in `Pipeline/ETL Notes.md`.

Key findings:
- The sync scheduler runs at 02:00 and 14:00 UTC daily
- GotSport is the sole upstream source for game results and roster data
- No dead-letter queue exists for failed API responses — failures are currently silently dropped and retried on the next scheduled run
- The webhook fallback listed in the v2 spec has never been implemented

### 2. GotSport API Rate Limit Audit
Pulled the last 7 days of sync logs from `Logs/gotsport-sync/`. GotSport appears to have reduced their rate limit from 120 req/min to approximately 60 req/min sometime in the last two weeks — no advance notice was found in their developer changelog. The current sync runner makes no accommodation for this and is hitting 429 errors during high-volume windows (primarily during the 02:00 UTC run when full roster refreshes are triggered).

- Affected runs: 6 of the last 14 scheduled syncs returned partial data
- 429 error rate during peak window: ~34% of requests
- Current retry logic: fixed 500ms delay, 3 attempts max — insufficient at the new limit

This is an active, ongoing problem. Escalated as a high-priority queue item (see Queue).

### 3. Stale Team Data Audit
Queried the `teams` table for records where `last_synced_at` is older than 72 hours. Found 847 teams with stale data — roughly 19% of the active team roster.

Root cause breakdown:
- ~540 teams: caught in failed sync windows caused by the rate limit issue above
- ~180 teams: belong to tournaments that ended and were de-prioritized by the sync scheduler (but are still marked `active`)
- ~127 teams: data simply missed in ingestion — unclear why, needs further investigation

Stale records have been logged to `Reports/stale-teams-2026-04-21.csv` for reference.
