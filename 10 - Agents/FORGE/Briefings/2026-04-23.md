### 1. SnapSoccer Source Research
**Deliverable:** `02 - Tiger Tournaments/Projects/Data Sources/SnapSoccer Source Research.md`

**Finding:** CONDITIONAL GO.

SnapSoccer is not a standalone platform — they run their competitive tournaments on **SincSports** (`soccer.sincsports.com`). This is the key architectural discovery. Their "Playdates" series (what USA Rank references repeatedly) are explicitly **score-free** by design — no results to scrape there. The value is in their ~10–15 competitive tournament events per year (Coastal Soccer Invitational, Coastal Academy Cup, Panama City Beach Classic, Gulf States Showcase, etc.), which DO have public scores on SincSports.

**Key numbers:**
- Estimated game volume: **10,000–20,000 historical games**
- Dedup risk: **LOW** (SincSports-native, no GotSport overlap)
- Auth required: **None** (fully public HTML pages)
- Dev effort: **6–12 hours** (adapts existing Sinc infrastructure)
- Source priority: **5** (same tier as Sinc Sports)
- Source ID format: `snapsoccer_{tid}_{gameId}`

Scraper skeleton is included in the memo. The scraper should crawl `snapsoccer.com/events/` to extract SincSports `tid=` parameters, then hit `soccer.sincsports.com/schedule.aspx?tid={TID}` for each competitive event. Playdate TIDs (`SNAPLEAG`, `SNAPLEAF`, `SSSPF`) must be blocklisted — no scores exist.

### 3. Daily Pipeline Cadence Scope
**Deliverable:** `02 - Tiger Tournaments/Projects/Product & Planning/Daily Pipeline Cadence Scope.md`

**Recommendation: Option A — Smart Overnight Batch with Active Team Filter.**

**Math verified:** 107,528 IDs × 1200ms ÷ 3 workers = 11.9 hours. Confirmed.

**Bottleneck:** The 1200ms self-imposed rate delay is the dominant cost. Reducing it is risky. Reducing the ID count is safe and achieves the same result.

**Active team filter:** SQL query targeting teams with games in last 90 days or scheduled in next 30 days yields ~15,000–40,000 IDs (15–35% of total, season-dependent). This cuts crawl time to **2.5–5 hours** — full pipeline completes by ~8 AM daily.

**Three options evaluated:**

| Option | Latency | Effort | Ships Before DSS? |
|--------|---------|--------|-------------------|
| A: Smart Overnight Batch | 24h | 4–7 hours | **Yes** |
| B: Segmented Daily Runs | 7-day team staleness | 6–10 hours | Yes |
| C: Event-Triggered Incremental | 2–6 hours | 20–40 hours | **No** |

Option A wins: minimal code change, no new infrastructure, ships in one session, matches USA Rank's effective cadence.

**Delivered:**
- `get-active-team-ids.sh` — SQL query → comma-separated ID list for `GSAPI_TEAM_IDS`
- `run-daily.sh` — full daily script (active crawl + pipeline)
- Crontab entries: `0 2 * * 1-6` (daily) + `0 23 * * 0` (weekly full sweep)
- Health check script for failed run detection
- Full rollback plan (5-minute revert to `run-overnight.sh`)

**Timeline:** Can be implemented and tested before May 1, shipping 2+ weeks before DSS.
