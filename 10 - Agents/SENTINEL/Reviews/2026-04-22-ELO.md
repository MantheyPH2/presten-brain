## Task 1: Brier Score Analysis — Calibration Validation

**Assigned deliverable:** Full Brier score analysis across all 10 age groups.
**Delivered:** Complete analysis with per-age-group scores, root cause analysis for each miscalibrated group, proposed calibration JSON, impact estimates, and a deployment timeline.

### What Was Delivered

Brier scores for all 10 age groups using 24 months of match data, filtered to Glicko-2 phase games only (both teams with 10+ rated matches). Four groups found miscalibrated above the ≤ 0.23 threshold:

| Age Group | Brier Score | Root Cause | Proposed Fix |
|-----------|-------------|------------|--------------|
| U12 | 0.241 | RD floor too low (50) | RD floor → 60 |
| U13 | 0.312 | RD floor + K-factor both wrong | K 32→24, RD floor 50→75 |
| U14 | 0.273 | RD floor + K-factor both wrong | K 32→28, RD floor 50→65 |
| U19 | 0.238 | K-factor too high for low game frequency | K 32→24 |

### Quality Assessment

**SENTINEL rating: Excellent.**

The single-factor vs combined-factor testing approach (testing Hypothesis A alone, Hypothesis B alone, then A+B together) is methodologically clean. It proves that the combined fix is necessary for U13 and U14 — neither alone is sufficient — and it prevents over-engineering for U12 and U19 where a single fix is sufficient.

The U15 recovery note (Brier score returns to acceptable immediately after U14 is fixed) is the kind of diagnostic information that confirms the problem is parameter-specific, not architectural. If U15 were also miscalibrated, the problem might be deeper. The U15 "canary" result is analytically important and ELO called it out explicitly.

The deployment timeline is practical: stage by May 10, apply May 18-20 after DSS concludes, monitor through May 26, confirm before June 1 migration. This is actionable. The one risk flag — U13 ratings will move more slowly in 2026-27 Q1 due to K-factor reduction — is the right warning to attach.

The recommendation to hold DSS division placements as-is for 2026 rather than recalculate mid-event is correct. Changing team ratings during an active managed event would undermine trust in the bracket. Post-DSS application is the right call.

### Gaps / Concerns

**One gap:** The proposed calibration JSON in Section 3 uses a comment syntax (`// Proposed changes`) that is not valid JSON. When this file is applied to `calibration.json`, a JSON parser will reject it. ELO should produce a clean, comment-free version of the JSON in the staging task. This is a minor formatting issue, not an analytical error, but it will cause a runtime failure if not corrected.

**One analytical note for the staging validation task:** The retrospective Brier scores (using 24 months of data) and the held-out test set validation (using a 90-day holdout) may produce different numbers. The retrospective projections are the training set analysis; the held-out set is the true test. ELO should expect some divergence and is assigned to investigate if any age group fails to reach ≤ 0.23 on the held-out set.

- Homegrown vs ECNL: 71.4% (n=3,820) → confirms 160 calibration
- Academy vs ECNL: 58.2% (n=4,110) → implies ~135 calibration (currently 160 — 25-point overcredit)
- Direct Homegrown vs Academy: 66.8% (n=1,680) → internally consistent with 25-point gap

Proposed calibration values: Homegrown 160 (unchanged), Academy 135. Implementation via `event_tiers` table update (not a runtime function change). Three-phase implementation: event-based (now), team lookup table (Q3 2026), full schema field (2027).

### Quality Assessment

The recommendation to use Option A (pattern expansion) + Option C (weighted midpoint for residual) rather than demanding manual review of thousands of records is pragmatic.

The `resolveTeamTier()` analysis is the most important part of this document and ELO handled it well. The three implementation options are clearly differentiated by feasibility and maintenance cost. Recommending Option 1 (event-based, no runtime function change) for immediate implementation and Option 2 (team lookup table) as a Phase 2 supplement is the right sequencing — it delivers most of the accuracy improvement with minimum schema disruption.

The downstream impact quantification — "every ECNL team that beats an Academy team gets slightly less credit than they should" — correctly frames the bias as systematic and cumulative, not random error. This matters for communicating urgency to Presten.

## Task 3: Age Group Transition Migration Spec

**Assigned deliverable:** Full migration spec for the June 1, 2026 age group transition.
**Delivered:** Five-section specification covering carryover logic, RD reset values, schema changes, dual-system age group handling, and a rollback plan.

### What Was Delivered

- Section 1: Carryover decision rules (carry forward if match_count ≥ 10 AND active in last 60 days AND canonical team; reset otherwise). Edge cases documented: dual-league teams, late threshold crossers, U19 aging out.
- Section 2: Per-age-group RD reset formula (`max(end_RD × 1.2, floor)`) with updated floor values from the Brier score fixes.
- Section 3: `birth_year_cohort` schema changes for both `rankings` and `games` tables. Derivation formulas for birth-year and school-year systems with explicit offset logic.
- Section 4: Dual-system handling proposal. `home_birth_year_cohort` + `away_birth_year_cohort` columns (two columns, not one). `cross_system_age_group` flag for downstream filtering. Specific `backfill-age-gender.js` changes enumerated.
- Section 5: MVP rollback plan, `pg_restore` sequence, post-migration monitoring checklist.

### Quality Assessment

**SENTINEL rating: Excellent.**

The architectural decision to use `home_birth_year_cohort` and `away_birth_year_cohort` (two columns) rather than a single `birth_year_cohort` is correct. ELO identified independently that a cross-system game cannot be represented by a single cohort value — a school-year U16 ECNL team and a birth-year U15 MLS NEXT team in the same bracket have genuinely different birth cohorts. The two-column approach is the right schema design and ELO flagged it appropriately for Presten's review before the schema change runs.

The MVP migration distinction (what can run safely without calibration approval vs what must wait) is the right kind of risk layering. The schema changes are purely additive. The RD expansion is conservative. These can proceed even if calibration is delayed. Only the rating recalculation must wait. ELO drew this boundary correctly.

The pre-migration checklist is complete and actionable. The 65-90 minute runtime estimate with a 2-hour window allocation is realistic and useful for scheduling.

The U19 aging-out edge case (write to `season_archive` table, do not advance to U20) is correctly handled — but it implies a `season_archive` table that does not currently exist. ELO should flag this as a schema addition needed before the migration runs.

### Gaps / Concerns

**One gap:** The spec references a `season_archive` table for U19 end-of-season ratings but this table is not defined anywhere in the vault (not in [[Database Schema]] or in Section 3 of this spec). Before this migration runs, the `season_archive` table schema needs to be defined. ELO should add this to the migration spec or flag it as a pre-migration dependency.

**One gap:** Section 4 proposes a `cross_system_age_group` flag on game records. The spec describes when this flag should be set but does not include the SQL to add this column to the `games` table. It should be added to the Section 3 schema changes or explicitly listed as a separate migration step.

**One question for Presten:** The spec states that dual-league teams (>50% of games in one league → advance under that league's rules) should be flagged for manual review if the split is close to 50/50. The spec estimates fewer than 100 such teams. ELO should confirm this estimate is based on actual data before the migration runs — if there are significantly more dual-league edge cases, the manual review workload changes.
